## Карта Supabase: где что лежит

Ниже не полный dump схемы, а рабочая карта самых важных таблиц и views.

### Ручной ввод

#### `manual_stats`

Главная таблица ручного ввода.

Хранит факты вида:

- `record_date`
- `channel`
- `account_name`
- `campaign_name`
- `metric_name`
- `value`

Используется так:

- `email`: ручные `booked_meetings`, `held_meetings`
- `linkedin`: ручные `booked_meetings`, `held_meetings`
- `app`: основной слой статистики
- `telegram`: основной слой статистики

Важно:

- исторически там есть legacy activity для `linkedin` (`connection_req`, `accepted`, `sent_messages`, `replies`)
- но в текущей целевой логике для `linkedin` manual activity не должна молча смешиваться с live Expandi activity

### TAL

Список TAL.

Хранит:

- id TAL

Связка:

### Expandi / LinkedIn

#### `expandi_accounts`

Список аккаунтов Expandi.

#### `expandi_campaign_instances`

Список campaign instances из Expandi.

Это live campaign layer.

Используется для понимания:

- какие кампании существуют сейчас
- в каком аккаунте они живут
- какие snapshot stats отдает Expandi

#### `expandi_campaign_catalog`

Исторический каталог campaign instances для `LinkedIn`.

Смысл:

- хранит все campaign instances, которые система когда-либо видела
- не только текущие live instances из Expandi
- нужен, чтобы отключение аккаунта или исчезновение кампании из live API не убивало historical message layer

Используется так:

- `expandi-sync` upsert-ит в него live campaign instances при каждом campaign sync
- message ingest валидирует `campaign_instance_id` уже по этому каталогу
- если historical catalog недоступен, sync временно падает назад на `expandi_campaign_instances`

Важно:

- это Phase 1 перехода `LinkedIn` из `catalog-first` в `history-first`
- таблица уже в live Supabase
- старую live table `expandi_campaign_instances` мы не удаляли

#### `expandi_campaign_stats_snapshots`

Snapshot layer из Expandi.

Хранит snapshot counters по campaign instance, например:

- `contacted_people`
- `connected`
- `replied_first_action`
- `replied_other_actions`

Это важный live-слой для `LinkedIn`, особенно для:

- `Connections`
- `Accepted`
- `Replies`

#### `expandi_campaign_daily_v`

Historical daily layer для LinkedIn.

Строится не из snapshot counters, а из message/history данных.

Нужен для:

- daily / range mode
- `sent_messages`
- historical replies/messages semantics

#### `expandi_kpi_daily_v`
#### `expandi_kpi_monthly_v`
#### `expandi_kpi_alltime_v`

Старый reporting layer для `LinkedIn`.

Важно:

- исторически он был основным источником страницы `LinkedIn`
- в нём были проблемы со смешиванием manual/legacy/live логики
- старые views пока не удалены

#### `linkedin_kpi_daily_v2`
#### `linkedin_kpi_monthly_v2`
#### `linkedin_kpi_alltime_v2`

Новый shadow/reporting layer для `LinkedIn`.

Идея:

- безопасно собрать более правильную логику без удаления старых views
- постепенно перевести UI на `v2`

Важно:

- legacy supplement в `v2` больше не берётся из всего `manual_stats(linkedin)` подряд
- в supplement попадают только кампании, подтверждённые в `linkedin_campaign_mapping` со статусом `legacy_only`
- `approved` и `group_only` не подмешивают manual activity в LinkedIn totals
- `ignored` полностью исключены из supplement

#### `linkedin_live_snapshot_v2`

Слой live snapshot-логики для `LinkedIn`.

Смысл:

- показывает текущие live-кампании из Expandi
- отражает live snapshot semantics

#### `linkedin_legacy_manual_activity_v2`

Legacy supplement layer для `LinkedIn`.

Смысл:

- хранит manual activity как отдельный supplement-слой
- нужен, чтобы не терять старую историю
- не должен автоматически и неявно смешиваться с live Expandi totals

### Как сейчас накапливается history в LinkedIn

Критичная логика для `Messages` и daily history:

1. `expandi-sync` тянет из Expandi:
   - accounts
   - campaign instances
   - messengers
   - messages
2. live campaign instances пишутся в `expandi_campaign_instances`
3. те же instances дублируются в `expandi_campaign_catalog`
4. `messengers/messages` принимаются в history слой только если их `campaign_instance_id` есть в historical catalog
5. дальше из `expandi_messages` строится `expandi_campaign_daily_v` и всё downstream reporting

Почему это важно:

- раньше whitelist был завязан только на текущий live слой `expandi_campaign_instances`
- из-за этого потерянный Expandi account мог ломать historical messages
- теперь для whitelist используется historical catalog, а не только текущий live snapshot

### Alias mapping

#### `campaign_name_aliases`

Таблица alias-нормализации названий кампаний.

Используется, когда:

- manual rows записаны под коротким именем
- live слой использует длинное canonical имя

Особенно важна для:

- `Email`
- `LinkedIn`

#### `linkedin_campaign_mapping`

Новый mapping layer для `LinkedIn`.

Хранит уже не только alias-нормализацию, а трёхступенчатую модель:

- `raw_name`
- `canonical_name`
- `campaign_group`

Зачем нужен:

- historical названия и live Expandi названия могут не совпадать
- `before / after / re-eng / conference` нельзя слепо склеивать в один canonical
- но иногда их нужно объединять в один более широкий `campaign_group`

Пример:

- `Affiliate World Bangkok before the conference`
- `Affiliate World Bangkok after the conference`

могут быть:

- разными `canonical_name`
- но одним `campaign_group = Affiliate World Bangkok`

Важно:

- current UI ещё не переведён на этот mapping layer
- это подготовка для безопасного `Phase 2` recovery и будущего `linkedin_kpi_*_v3`
- таблица additive, ничего текущего не ломает
- в ней есть `match_status`:
  - `approved`
  - `needs_review`
  - `group_only`
  - `legacy_only`
  - `ignored`

#### `linkedin_campaign_mapping_conflicts_v`

Контрольный view для антидублей и опасных маппингов.

Ловит как минимум 3 класса проблем:

- `raw_name` совпадает с live Expandi campaign name, но маппится в другой `canonical_name`
- `canonical_name` отсутствует в `expandi_campaign_catalog`
- один `campaign_group` содержит несколько разных `canonical_name`

Правило:

- перед ручным добавлением legacy-названий нужно смотреть этот view
- если там есть конфликт, такой mapping нельзя без review пускать в recovery/TAL слой
- явно неверные auto-mappings (`before -> after`, generic alias на live campaign) нужно чистить отдельной migration, а не править “на глаз” в UI
- названия из Google Sheet по умолчанию лучше сначала seed-ить как `needs_review`, а не сразу пускать в approved recovery path
- после seed можно отдельной migration переводить только очевидные live matches в `approved`, сохраняя:
  - `canonical_name` = точная live Expandi кампания
  - `campaign_group` = удобная бизнес-группа для TAL
- `group_only` использовать для общих business-labels вроде `START Summit`, `Esports`, `Payroll services`, когда:
  - label нужен для TAL/grouping
  - но его нельзя использовать как отдельный KPI supplement без риска дубля
- `legacy_only` использовать для подтверждённых исторических кампаний, которых уже нет в live `expandi_campaign_catalog`, но по ним есть реальный legacy-след в Supabase
  - пример: `AdTech`
  - такие строки не надо автоматически матчить к live Expandi campaigns
  - но их можно использовать в controlled recovery historical KPI
- `ignored` использовать для названий, которые продуктово решили вообще не учитывать в LinkedIn totals / recovery
  - текущий пример: `Travel Tech`

Текущее состояние LinkedIn mapping после чистки sheet-seed:

- obvious live matches уже переведены в `approved`
- общие business labels переведены в `group_only`
- подтверждённые legacy-only строки переведены в `legacy_only`
- открытых `needs_review` по текущему LinkedIn sheet-seed больше не осталось
- нулевые или продуктово исключённые sheet-labels выведены в `ignored` и не участвуют в LinkedIn recovery / KPI:
  - `Coinfest`
  - `Future Travel`
  - `MWC`
  - `Partnership`
  - `SBC Conference`
  - `Sigma Rome`
  - `Ticketing Platforms`
  - `Travel Tech Asia`
  - `Warsaw AI`
  - `Travel Tech`

## Как устроена система в целом

Есть несколько независимых, но связанных слоёв:

1. `portal` на Next.js
- рендерит dashboard'ы и формы
- читает Supabase напрямую и через internal API routes

2. `Supabase`
- хранит таблицы, views, миграции
- это canonical storage layer

3. sync-функции / интеграции
- подтягивают данные из `SmartLead`, `Expandi` и др.
- складывают raw и агрегированные данные в Supabase

4. manual input
- часть бизнес-метрик вносится вручную через `manual_stats`

## Важные продуктовые сущности

### `manual_stats`

Это таблица ручного ввода.

Используется для:

- `email`: `booked_meetings`, `held_meetings`
- `linkedin`: `booked_meetings`, `held_meetings`
- `app`: основная статистика канала
- `telegram`: основная статистика канала

Важно:

- `manual_stats` уже укреплён:
  - введён `upsert`
  - введена логическая уникальность факта
- повторное сохранение одного и того же факта не должно раздувать KPI

### `TAL`

TAL сейчас агрегирует:

- `Email`
- `LinkedIn`
- `App`
- `Telegram`

Как считается:

- внутри одного канала несколько кампаний суммируются
- между каналами KPI не смешиваются в один общий outreach-счётчик
- у каждого канала свой блок метрик
- `total_meetings` = сумма `booked_meetings` по всем включённым каналам
- `total_held_meetings` = сумма `held_meetings` по всем включённым каналам

Источники:

- `Email` -> `smartlead_kpi_alltime_v`
- `LinkedIn` -> `linkedin_kpi_alltime_v2`
- `App` -> `app_kpi_alltime_v`
- `Telegram` -> `telegram_kpi_alltime_v`

Важно:

- в `tal_campaigns` теперь есть `source_campaign_key`
- новые TAL работают по модели `id-first, name-fallback`
- `SmartLead` уже использует нативный `campaign_id`
- `App` и `Telegram` уже используют внутренние стабильные ключи по нормализованному имени
- `LinkedIn` теперь использует внутренний canonical source key вида `expandi:canonical:<normalized campaign name>`
- это сделано потому, что в `expandi_campaign_instances` нативный `campaign_id` по факту `null` и не годится как stable id для TAL
- старые и новые LinkedIn-строки в `tal_campaigns` уже бэкфиллятся в этот canonical key, поэтому TAL больше не зависит от account-aware split
- rename/alias drift для LinkedIn всё ещё возможен, если поменяется сам canonical campaign name, но зависимость от raw account rows уже убрана
- но LinkedIn TAL теперь смотрит на тот же `v2` слой, что и страница `/expandi`

Основные таблицы/слой:

- `tals`
- `tal_campaigns`
- `tal_analytics_v`

## Принципы безопасных правок

Нельзя делать:

- удаление старых таблиц/views без переходного слоя
- silent rewrite существующих боевых views без возможности rollback
- массовое destructive изменение данных без backup

Нужно делать:

1. Сначала additive change
- новая таблица
- новая view
- новая migration

2. Потом shadow check
- сравнить старую и новую логику

3. Потом переключение UI

4. Потом cleanup legacy, если он реально больше не нужен

## Что проверять после любой правки

Минимальный smoke-check:

2. Основные KPI не пустые
3. `By Campaign` и верхний блок не расходятся очевидно
4. Ручной ввод сохраняется и не дублирует факт
5. Если правка касается `TAL`, нужно проверить:

Если задача касается цифр:

1. Сначала понять, какой канал
2. Потом понять, это:

### Новые SQL views

#### `smartlead_replies_v`

Что это:

Browseable view всех email-ответов.

Что показывает:

Что важно:

- добавлен индекс `smartlead_leads_campaign_lead_id_idx` для быстрого join
- `message_body` в `smartlead_events` содержит ИСХОДЯЩИЙ шаблон, а НЕ текст ответа - это ограничение API `/api/v1/campaigns/{id}/statistics`

#### `expandi_replies_v`

Что это:

Browseable view всех LinkedIn-ответов.

Что показывает:

- дату и время ответа
- LinkedIn аккаунт (имя)
- кампанию
- контакт (имя, email, компания, должность, profile link)
- текст ответа (body)
- привязку к TAL

Источник:

- `expandi_messages` WHERE `is_inbound = true`
- JOIN `expandi_messengers` для contact info
- LEFT JOIN `expandi_campaign_instances` для campaign name
- LEFT JOIN `expandi_accounts` для account name
- LEFT JOIN `tal_campaigns` + `tals` для TAL

Что важно:

- добавлен индекс `expandi_messages_inbound_received_idx` (partial, filtered by is_inbound=true)
